# 다바르(DABAR) 주간 코드 점검 리포트

**날짜:** 2026-06-04  
**점검자:** 자동화 AI 점검 시스템  
**대상:** 다바르 성경 퀴즈 웹 앱 (Next.js · Supabase · Claude API)

---

## 요약

전반적으로 앱의 핵심 기능(퀴즈 진행, 문제 생성)은 잘 구성되어 있으나, **API 입력값 검증 누락**과 **오류 처리 미흡**이 실서비스 품질에 영향을 줄 수 있습니다. PWA(앱처럼 설치하기) 기능이 아이콘 파일 부재로 동작하지 않으며, 타이머 관련 버그가 사용자 점수 계산에 영향을 줄 수 있습니다. 즉시 해결이 필요한 항목 4건, 단기 개선 항목 5건, 중장기 기능 개선 3건을 확인했습니다.

---

## 1. 발견된 버그 및 오류

### 🔴 [심각도: 높음] 타이머 종료 시 점수 계산 오류
- **파일:** `app/quiz/page.tsx` (50~51번 줄)
- **설명:** 30초 타이머가 0이 되어 자동으로 다음 문제로 넘어갈 때, 점수가 타이머가 시작된 시점의 값으로 고정되어 이후 맞춘 문제의 점수가 반영되지 않을 수 있습니다.
- **사용자 영향:** 실제로 맞춘 문제 수보다 낮은 점수가 결과 화면에 표시될 수 있습니다.

### 🟠 [심각도: 중간] 네트워크 오류 시 안내 없음
- **파일:** `app/quiz/page.tsx` (33~34번 줄)
- **설명:** 인터넷 연결 문제나 서버 오류로 문제를 불러오지 못할 경우, 사용자에게 아무런 안내 없이 "문제가 없습니다."라는 메시지만 표시됩니다.
- **사용자 영향:** 사용자가 앱 오류인지, 실제로 문제가 없는 건지 알 수 없어 혼란을 겪습니다.

### 🟠 [심각도: 중간] 결과 페이지 직접 접근 시 무한 로딩
- **파일:** `app/result/page.tsx` (21~22번 줄)
- **설명:** 퀴즈를 풀지 않고 결과 페이지 주소를 직접 입력하거나 브라우저를 새로고침하면 "로딩 중..."이 계속 표시되고 멈추지 않습니다.
- **사용자 영향:** 결과를 확인하려던 사용자가 빈 화면에 갇히게 됩니다.

### 🟡 [심각도: 낮음] 결과 데이터 손상 시 앱 충돌
- **파일:** `app/result/page.tsx` (21번 줄)
- **설명:** 브라우저의 임시 저장 공간(sessionStorage)에 저장된 퀴즈 결과 데이터가 손상되었을 경우, 안전 처리 코드 없이 그대로 읽어 앱이 흰 화면으로 충돌할 수 있습니다.
- **사용자 영향:** 드물지만 발생 시 앱을 완전히 종료하고 재시작해야 합니다.

---

## 2. 미완성 또는 작동하지 않는 기능

### 🔴 PWA 아이콘 파일 없음 (앱 설치 불가)
- **파일:** `public/manifest.json`, `public/icons/` 디렉토리
- **설명:** 앱 설정 파일에는 홈 화면 추가(앱처럼 설치)에 필요한 아이콘 2종(192×192, 512×512 픽셀)이 지정되어 있지만, 실제 이미지 파일이 존재하지 않습니다.
- **사용자 영향:** 스마트폰/태블릿 홈 화면에 앱을 추가할 수 없으며, 일부 브라우저에서 PWA 설치 안내가 표시되지 않습니다.

### 🟡 퀴즈 중 새로고침 시 처음부터 시작
- **설명:** 퀴즈를 풀다가 실수로 페이지를 새로고침하면 진행 상황이 모두 사라지고 처음부터 다시 시작됩니다.
- **사용자 영향:** 여러 문제를 풀다가 실수로 새로고침하면 진행 상황을 잃게 됩니다.

### 🟡 이전 퀴즈 설정 기억 안 됨
- **설명:** 앱을 닫았다가 다시 열면 이전에 선택했던 난이도, 성경 구분, 문제 수가 초기화됩니다.
- **사용자 영향:** 자주 사용하는 사용자가 매번 같은 설정을 반복해야 합니다.

### 🟡 점수 기록/통계 기능 없음
- **설명:** 지금까지 풀었던 퀴즈 점수 기록이나 "어떤 책이 약한지" 같은 통계 기능이 없습니다.
- **사용자 영향:** 학습 진도를 추적하거나 약한 부분을 파악하기 어렵습니다.

## 3. 보안 취약점

### 🔴 [심각도: 높음] API 요청 수 제한 없음 — 과부하 공격 취약
- **파일:** `app/api/questions/route.ts` (13, 20번 줄)
- **설명:** 문제를 불러오는 서버 기능에 요청 수 제한이 없습니다. 악의적인 사용자가 `count=999999`처럼 비정상적인 값을 전달하면 서버가 수십만 건의 데이터를 조회하려고 시도하고, 반복 요청으로 서비스가 느려지거나 멈출 수 있습니다.
- **권고:** `count` 값을 최소 1, 최대 30으로 제한하는 코드 1줄 추가만으로 해결됩니다.

### 🟠 [심각도: 중간] 요청 횟수 제한(Rate Limiting) 없음
- **파일:** `app/api/questions/route.ts`
- **설명:** 같은 IP에서 짧은 시간 내 수백~수천 번 요청을 반복해도 차단되지 않습니다. Supabase 무료 플랜의 경우 일일 요청 한도를 소진시킬 수 있습니다.
- **권고:** Vercel Edge Config 또는 Upstash Redis를 이용한 간단한 횟수 제한 추가를 권장합니다.

### 🟡 [심각도: 낮음] 책 이름 입력값 검증 없음
- **파일:** `app/api/questions/route.ts` (16~23번 줄)
- **설명:** URL에서 전달받은 성경 책 이름을 유효한 66권 목록과 대조하지 않고 DB 쿼리에 사용합니다. Supabase의 안전한 쿼리 방식 덕분에 데이터 변조는 불가능하지만, 잘못된 값이 그대로 데이터베이스에 전달됩니다.
- **권고:** `lib/bible.ts`에 있는 책 이름 목록으로 입력값을 검증하는 코드 추가를 권장합니다.

### 🟡 [심각도: 낮음] 관리자 DB 클라이언트 불필요한 노출
- **파일:** `lib/supabase.ts` (8~11번 줄)
- **설명:** 데이터베이스 관리 권한을 가진 `supabaseAdmin` 클라이언트가 공용 라이브러리 파일에 정의되어 있습니다. 현재는 문제가 없지만, 실수로 클라이언트 측 코드에서 이를 불러올 경우 관리자 권한이 노출됩니다.
- **권고:** 관리자 클라이언트를 서버 전용 파일(`lib/supabase.server.ts`)로 분리를 권장합니다.

## 4. 성능 개선 포인트

### 🟠 DB에서 필요 이상으로 많은 데이터 조회
- **파일:** `app/api/questions/route.ts` (20, 29번 줄)
- **설명:** 10문제를 요청하면 서버는 50개(10 × 5)를 DB에서 가져온 뒤 40개를 버립니다. 30문제를 요청하면 150개를 가져옵니다. DB에서 직접 무작위로 N개를 선택하는 방식(`ORDER BY RANDOM()`)이 훨씬 효율적입니다.
- **절약 효과:** DB 조회량 최대 80% 감소 예상.

### 🟡 무작위 셔플 알고리즘 품질 문제
- **파일:** `app/api/questions/route.ts` (29번 줄)
- **설명:** 현재 사용 중인 무작위 섞기 방식은 수학적으로 완벽하게 균등하지 않아 특정 문제가 더 자주 또는 덜 나올 수 있습니다. Fisher-Yates 알고리즘으로 교체를 권장합니다.

## 5. 출시/운영 관점 우선순위 제안

# DABAR 주간 점검 리포트 — 2026-06-05

> 다바르(DABAR) 성경 퀴즈 앱의 코드를 한 주간 점검한 결과입니다.
> 개발 지식이 없어도 이해하실 수 있도록 풀어서 정리했습니다.
> **이 리포트는 점검 결과만 담고 있으며, 코드는 수정하지 않았습니다.**

## 1. 한눈에 보기

| 구분 | 개수 | 한 줄 요약 |
|------|:---:|------|
| 🔴 꼭 고쳐야 할 문제 | 3건 | 화면 멈춤, 앱 아이콘 누락, 악용 가능한 입력값 |
| 🟡 미완성·정리 필요 | 5건 | 안 쓰는 코드/설정, 실행 방법 불일치 |
| 🟢 더 좋게 만들 점 | 6건 | 보기 섞기, 오류 안내, 접근성 등 |

**총평:** 앱의 핵심 기능(문제 선택 → 풀이 → 결과)은 잘 동작하도록 짜여 있습니다.
다만 **출시(특히 휴대폰 앱 설치) 전에 반드시 손봐야 할 부분**이 몇 군데 있습니다.
가장 시급한 것은 **① 앱 아이콘 파일 누락**과 **② 인터넷이 끊겼을 때 화면이 멈추는 문제**입니다.

## 2. 점검한 범위

- 사용자 화면: 첫 화면(문제 고르기), 퀴즈 풀이 화면, 결과 화면
- 서버 기능: 문제를 불러오는 API
- 데이터베이스 설계(Supabase) 및 보안 규칙
- 문제 자동 생성 스크립트(AI로 성경 퀴즈를 만들어 채워 넣는 도구)
- 앱 설정 파일들(PWA 설치 설정, 빌드 설정 등)

## 3. 발견된 문제 (우선순위 순)

### 🔴 A. 휴대폰 앱 아이콘 파일이 없음 — *출시 직결*
- **무슨 일?** 앱을 휴대폰 홈 화면에 설치할 때 쓰는 아이콘 그림 파일(`icon-192.png`, `icon-512.png`)이
  실제로는 저장소에 없습니다. 그런데 설정 파일(`manifest.json`)과 화면 설정(`layout`)은
  "이 그림을 쓰겠다"고 가리키고 있습니다.
- **그래서 뭐가 문제?** 사용자가 앱을 홈 화면에 추가하면 **아이콘이 깨져 보이거나 빈 칸으로** 나옵니다.
  앱의 첫인상을 크게 해칩니다.
- **권장:** 192×192, 512×512 크기의 아이콘 이미지를 만들어 `public/icons/` 폴더에 넣기.

### 🔴 B. 인터넷이 끊기면 "문제를 불러오는 중..." 화면에서 멈춤
- **무슨 일?** 퀴즈 화면에서 서버에 문제를 요청할 때, **요청이 실패하는 경우(인터넷 끊김, 서버 오류 등)에
  대한 대비가 없습니다.**
- **그래서 뭐가 문제?** 통신이 잘 안 되면 화면이 "문제를 불러오는 중..."에 **영원히 멈춰**
  사용자는 무엇이 잘못됐는지 알 수 없습니다. 지하철·약한 와이파이 환경에서 충분히 발생합니다.
- **권장:** 요청 실패 시 "문제를 불러오지 못했어요. 다시 시도해 주세요" 같은 안내와 재시도 버튼 추가.

### 🔴 C. 문제 개수 입력값에 제한이 없음 — *비용·악용 위험*
- **무슨 일?** 문제를 불러오는 주소(`/api/questions`)에 "문제 수(count)"를 숫자로 받는데,
  **상한선이 없습니다.** 정상 화면에서는 5~30개만 고르게 돼 있지만, 주소창을 직접 고치면
  누구나 `count=100000` 같은 값을 넣을 수 있습니다.
- **그래서 뭐가 문제?** 그 값의 5배(`count × 5`)만큼 데이터베이스에서 한꺼번에 긁어오려 시도하므로,
  **서버·데이터베이스 부하가 커지고 비용이 증가**할 수 있습니다. 숫자가 아닌 값(`count=abc`)이
  들어오면 동작이 불안정해질 수도 있습니다.
- **권장:** 서버에서 count를 **1~50 사이로 강제 제한**하고, 숫자가 아니면 기본값(10)으로 처리.

---

### 🟡 D. "안 쓰는데 설치만 돼 있는" 부품들 (정리 권장)
1. **Tailwind CSS 미사용:** 디자인 도구인 Tailwind/PostCSS/Autoprefixer가 설치돼 있지만,
   실제 화면은 전부 직접 스타일을 써서 만들었습니다. **즉, 설치만 해두고 안 씁니다.**
   설정 파일(`tailwind.config`, `postcss.config`)도 없습니다. → 빼거나, 쓸 거면 설정 추가.
2. **관리자용 DB 접속 코드가 쓰이지 않음:** `lib/supabase.ts`의 `supabaseAdmin`(강력한 권한의 열쇠를 쓰는
   접속)이 만들어져 있지만 **어디서도 사용하지 않습니다.** 보안상 위험을 줄이기 위해 안 쓰는 건 지우는 게 좋습니다.
3. **안 쓰는 함수:** `lib/bible.ts`의 `booksForTestament` 함수가 만들어져 있지만 화면에서는 쓰지 않습니다.

### 🟡 E. 문제 생성 도구의 "실행 방법"이 두 가지로 엇갈림
- 사용 설명서(`package.json`)에는 `ts-node`로 실행하라고 돼 있는데,
  실제 클라우드 자동 실행(GitHub Actions)에서는 **`ts-node`가 동작하지 않아 `tsx`로 바꿔** 돌리고 있습니다.
- **그래서 뭐가 문제?** 새로 합류한 담당자가 설명서대로 `npm run generate`를 돌리면 **실패**할 수 있습니다.
- **권장:** 설명서의 실행 명령도 `tsx`로 통일.

### 🟡 F. 화면 설정의 "구버전 방식" 경고
- 앱 화면 기본 설정(`layout.tsx`)에서 테마 색상·화면 배율 설정을 **현재 Next.js 14가 권장하지 않는 옛 방식**으로
  넣어 두었습니다. 당장 큰 문제는 아니지만 **빌드할 때마다 경고 메시지**가 나오고,
  나중에 버전을 올리면 동작하지 않을 수 있습니다. → 권장 방식(`viewport` 설정)으로 옮기기.

### 🟡 G. 결과 데이터가 깨졌을 때 대비 없음
- 퀴즈 결과는 브라우저에 잠시 저장됐다가 결과 화면에서 다시 읽어옵니다. 이때 **저장된 내용이 손상되면
  결과 화면이 멈출 수 있습니다.** 흔하진 않지만 안전장치(오류 시 홈으로 보내기)를 넣어두면 좋습니다.

---

### 🟢 H. 더 좋게 만들면 좋은 점들
1. **보기(선택지) 순서 섞기:** 지금은 4개 보기의 순서가 고정입니다. AI가 만든 문제에서 정답이
   특정 위치에 몰리면 **눈치로 맞히기** 쉬워질 수 있습니다. 매번 보기 순서를 섞어주는 것을 권장합니다.
2. **문제 섞는 방식 개선:** 현재 문제를 무작위로 섞는 방식이 통계적으로 고르지 않습니다.
   더 균등한 방식(피셔-예이츠)으로 바꾸면 같은 문제만 자주 나오는 현상을 줄일 수 있습니다.
3. **중복 접속 코드 정리:** 서버 API가 데이터베이스 접속을 따로 또 만들고 있습니다(`lib`에 이미 있는데도).
   하나로 합치면 관리가 쉬워집니다.
4. **접근성(화면 확대 허용):** 현재 설정이 **손가락으로 화면을 키우는(확대) 동작을 막고** 있습니다.
   시력이 약한 분들을 위해 확대를 허용하는 것을 권장합니다.
5. **오프라인 지원:** PWA 설치 설정은 있지만, 인터넷 없이도 쓰게 해주는 기능(서비스 워커)은 없습니다.
   "설치형 앱"의 장점을 살리려면 추가를 고려.
6. **점수 기록·랭킹 없음:** 지금은 한 판이 끝나면 결과가 사라집니다. 사용자 재방문을 늘리려면
   "내 기록 저장", "친구와 점수 공유", "주간 랭킹" 같은 요소가 효과적입니다.

## 4. 출시·운영 관점 — 다음에 하면 좋은 일 (우선순위)

> **1순위는 "출시를 막거나 첫인상을 해치는" 것**, 뒤로 갈수록 "있으면 좋은" 것입니다.

| 순위 | 할 일 | 이유 | 예상 난이도 |
|:---:|------|------|:---:|
| **1** | 앱 아이콘 이미지 추가 (이슈 A) | 휴대폰 설치 시 깨진 아이콘 — 출시 직결 | 낮음 |
| **2** | 문제 불러오기 실패 시 오류 안내·재시도 (이슈 B) | 약한 통신 환경에서 화면 멈춤 방지 | 낮음 |
| **3** | 문제 개수 입력값 제한 (이슈 C) | 서버 비용·악용 방지 | 낮음 |
| **4** | 보기 순서 섞기 (이슈 H-1) | 퀴즈 난이도/공정성 — 사용자 체감 품질 | 낮음 |
| **5** | 실행 방법 통일 + 구버전 설정 정리 (E, F) | 운영·유지보수 혼선 제거 | 낮음 |
| **6** | 안 쓰는 부품 정리 (이슈 D) | 보안 위험·앱 용량 줄이기 | 중간 |
| **7** | 점수 기록·공유·랭킹 (이슈 H-6) | 재방문·입소문 — 성장 동력 | 높음 |
| **8** | 오프라인 지원·접근성 개선 (H-5, H-4) | 앱 완성도·사용자층 확대 | 중간 |

### 운영 측면 추가 제안
- **문제 자동 생성 도구 보안:** 로컬 설정 스크립트(`setup-cloud-generate.sh`)가
  API 키를 `.env.local`에서 읽어 GitHub에 등록하고 **`main`에 바로 밀어 넣습니다.**
  지금은 1회용 도구라 괜찮지만, 키가 외부에 노출되지 않도록 **주기적으로 키를 교체**하는 습관을 권장합니다.
- **AI가 만든 문제의 품질 검수:** 생성 스크립트는 형식(보기 4개, 정답 번호 등)은 검사하지만
  **내용이 신학적으로 맞는지는 사람이 확인하지 않습니다.** 출시 전 표본 검수 절차를 두면 좋습니다.
- **테스트가 전혀 없음:** 자동으로 동작을 확인하는 테스트가 없어, 수정할 때마다
  다른 곳이 깨졌는지 사람이 일일이 확인해야 합니다. 핵심 흐름부터 간단한 테스트를 추가하면 안정적입니다.
- **README(사용 설명서) 없음:** 새 담당자가 프로젝트를 이해하고 실행하는 데 시간이 걸립니다.

## 5. 참고 — 잘 되어 있는 점
- 데이터베이스 보안 규칙(RLS)이 켜져 있고, **읽기만 허용**돼 있어 일반 사용자가 문제를 수정할 수 없습니다. 👍
- 문제 생성 도구가 **중간에 끊겨도 이어서** 만들 수 있고, **이미 만든 문제는 건너뛰며**, 비용 한도(`--limit`)도
  조절할 수 있게 잘 설계돼 있습니다. 👍
- 화면 구성과 색상 테마가 한 곳(`theme.ts`)에 정리돼 있어 디자인 일관성이 좋습니다. 👍
